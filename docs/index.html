Maestro documentation

An open-source, offline-capable, no-code test-automation platform for automotive Hardware-in-the-Loop (HIL) and embedded device testing.

Maestro lets mostly-manual engineers design tests visually on a canvas, run them against automotive targets (Linux / QNX / Android attached to Windows or RDP host machines), and get detailed, step-level reports with attachments. Device control lives in your own tools — Maestro only orchestrates them through thin adapters.

Visual, not code

Drag actions onto a React Flow canvas; branches, parallel lanes and endurance loops — no Python required.

Your tools

Adapters wrap turboadb, turbossh and your bench scripts. Maestro never reimplements adb/ssh/power.

Real reports

Genuine Allure results (with trends) + JUnit XML, with inline logs, screenshots and DLT attachments.

Offline

Runs identically on a shared server and a bench laptop, fully offline after install.

Design principles

  1. Manual-engineer first. The primary path is a visual canvas + template gallery. Writing Python is an optional escape hatch, never required.
  2. Scenarios are data. A test case is a versioned list of steps that a single generic runner executes — not per-test Python files.
  3. One source of truth per capability. Device logic lives in your tools (turboadb, turbossh, power/DLT scripts); Maestro only calls them through thin adapters.
  4. Drop-in extensibility. Any script you write becomes a runnable step with zero framework changes; new adapters are auto-discovered from a manifest.
  5. Open-source only. No paid licenses, certificates or internet required at run time (Jira/Xray and email are optional outbound integrations).

Architecture

One React single-page app over a FastAPI backend, with a pluggable adapter layer that drives the bench.

LayerResponsibilityBuilt on
FrontendVisual designer, management screens, live run, reports. Served by the backend in production.React 18, Vite, TypeScript, Tailwind, React Flow
Backend / APIPersistence, ownership, scheduling, run orchestration, live WebSocket events, integrations.FastAPI, SQLAlchemy 2, Pydantic v2, APScheduler
Execution engineWalks a test case's steps (serial / parallel / step / endurance), dispatches each to an adapter, records results + attachments.async executor
AdaptersThin, manifest-discovered plugins; each is the only place that knows one tool's interface.turboadb, turbossh, your scripts, ffmpeg
StorageSQLite on a laptop, PostgreSQL on a server — same code, connection string only.SQLAlchemy + AES-256-GCM vault

Each adapter exposes a set of named actions (async functions returning a uniform AdapterResult). A test step is just { "action": "ssh.execute_command", "parameters": {…}, "timeout_seconds": 30 }.

Visual designer

The designer is a real React Flow graph canvas (the default ✦ Flow view; a Classic canvas and a List view are also available). You pick a Suite → Scenario → Test case, then drag actions from the categorized, searchable palette onto the canvas.

Nodes

  • Each node is one step: an action chip, a label, a timeout chip, and a "no target" warning if it needs a device binding.
  • Add by dragging from the palette or the "Add node" button. Edit by double-clicking. Hover a node for edit / branch / delete.
  • Drag a node up or down to change run order; "Tidy" auto-arranges top-to-bottom. A minimap and zoom controls keep large flows readable.

Run order & targets

Run order is derived from the canvas layout (top → bottom). Each step can be bound to a configured device target (its decrypted credentials are injected at run time), or left local. The Run on selector chooses Local or a saved RDP machine.

Branches, parallel & loops

Branches (Yes / No)

Turn a node into a decision with its branch button: it checks the previous step's output for a string and routes to one node when it matches and another when it doesn't. On save, the canvas's stable node ids are compiled into the engine's step-number jumps, so branches survive re-ordering.

Parallel

Group adjacent steps so they run concurrently, then rejoin — useful for capturing a screenshot while a log streams.

Endurance loops

Repeat a whole test case for N cycles with stop conditions — see Endurance / stability.

Step modes & control-flow knobs

KnobEffect
_loopRepeat a single step N times (all iterations must pass).
_if{source_step, contains, skip_to, else_skip_to} — jump based on an earlier step's output.
_parallel_groupConsecutive steps sharing a group name run concurrently.
_always_runCollector steps (logs/screenshots) that run even after an earlier failure.
device_config_idBind the step to a configured device; credentials are merged + the device is locked for the step.
{{steps.N.output}}Substitute a previous step's output into a string parameter.

Run your own scripts

There is no special folder and no framework change. Keep your scripts wherever they live and point a step at them. Output streams live to the run console, and produced files attach to the report.

StepUse it for
system.run_fileAny script/exe by path, with your own args.
system.run_commandAny full command line (through the shell).
system.run_registeredA script you registered once under Templates → Scripts, run by script_id.
system.run_scriptInline Python/PowerShell you paste in (sandboxed).

Interpreter is auto-detected

# by file extension
.ps1  -> powershell -NoProfile -ExecutionPolicy Bypass -File <file> <args>
.bat  -> cmd /c <file> <args>
.cmd  -> cmd /c <file> <args>
.py   -> python <file> <args>     # set python_path to override
.sh   -> bash <file> <args>
.exe  -> <file> <args>             # run directly

Parameters (all optional)

{
  "path": "C:/bench/power_control.py",
  "args": ["normal_power_cycle", "--port", "COM5"],   // list or string
  "cwd": "C:/bench",                  // working directory
  "env": { "TARGET": "ECU1" },          // extra environment vars
  "timeout": 120,                       // seconds
  "expect_contains": "OK",               // pass/fail on stdout (match_mode: contains|wildcard|regex)
  "attach_output": true,                // attach stdout to the report
  "match_file": "C:/bench/out/run.dlt",  // + match_pattern: pass only if a regex is found in a produced file
  "match_pattern": "BootComplete"
}

The bench-tool adapters (power.*, dlt.*, etfw.*) are the same idea with a fixed verb set — set script_path on the device config (plus com_port / channel / extra_args). With no script configured they simulate, so test design works without a bench.

Custom adapter plugins

To turn a script into a reusable action available everywhere, drop a folder under data/plugins/<name>/ with a manifest and an adapter:

# manifest.json
{ "name": "myscript", "version": "1.0", "entry_point": "adapter:MyScriptAdapter" }

# adapter.py
from backend.adapters.base_adapter import AdapterResult, BaseAdapter

class MyScriptAdapter(BaseAdapter):
    name = "myscript"
    def _register_actions(self):
        self.actions = {"do_thing": self._do_thing}
    async def _do_thing(self, params):
        return await self._run_subprocess(["python", r"C:\x.py"], timeout=60)

Restart, or use Plugins → Reload. Your actions appear as myscript.do_thing, ready to drop on the canvas.

Commands, file transfer and log capture on Linux / QNX targets and Windows / RDP boxes via turbossh — it handles domain (DOMAIN\user) and UPN logins, legacy ECU algorithms, host-key policy and jump hosts. Plus webcam and remote-desktop capture.

Power · DLT · ETFW

Thin wrappers around your bench scripts. Set script_path on the device config; with none set they simulate.

Power

Verbs match your power_control.py: normal_power_cycle, edl_power_cycle, power_off_edl_off, power_on_edl_off, power_on_edl_on, plus power_on, power_off, status, run (any subcommand) and check (validate the script without toggling power — used as the connection probe).

DLT

start_capture / stop_capture / capture drive your dlt.py (or a built-in TCP capture when no script is set); verify_pattern (regex over the trace), tail, save_file (attach a .dlt).

ETFW

bus_sleep_on, bus_sleep_off, set_state <state>, get_state (last state is tracked for assertions).

System · Serial · Camera

System

wait, echo, assert_contains, run_file, run_command, run_registered, run_script, fail — the no-hardware glue and the script runners (see Run your own scripts).

Serial

list_ports, wait_for_pattern (boot/login), monitor (capture the console), send — COM port + baud.

Camera (local)

screenshot (desktop), capture / capture_webcam (still), record_video, list_devices, detect. Needs ffmpeg (see Configuration).

Reports — Allure & JUnit

Every run produces an HTML report with collapsible step cards, parameters, full output, retries and inline attachments (screenshots embedded, logs/DLT downloadable). Beyond that, Maestro exports the real interchange artifacts:

  • Real Allure results — GET /api/reports/{id}/allure downloads an allure-results bundle. Run allure serve <dir> (or allure generate) for the genuine Allure UI: overview, timeline, graphs, categories and trends/history across runs (a stable history id lines runs of the same case up).
  • JUnit XML — GET /api/reports/{id}/junit for CI dashboards and Jira/Xray import.
  • Server-rendered Allure HTML — GET /api/reports/{id}/allure-html if the Allure CLI is installed on the server.

From the Reports page you can open/download a report, download Allure results or JUnit, compare two runs, bulk-delete, and re-run.

Endurance / stability

Run a whole test case repeatedly to find intermittent failures. Set a cycles count in the Run menu (or cycles + stop_conditions on POST /api/executions).

Stop conditionMeaning
max_durationStop after this many seconds of wall-clock time.
consecutive_failuresStop after N cycles fail in a row.
failure_thresholdStop after N total failed cycles.

Each cycle records a CycleResult; the Reports detail shows a roll-up — per-cycle pass/fail chips, totals, and the first-failure cycle. GET /api/reports/{id}/cycles returns the raw data. Pairs naturally with Allure trends.

Integrations

Pluggable result publishers, all optional and config-driven via environment variables; unconfigured channels are a safe no-op.

ChannelSendsConfigure
EmailAn HTML run summary + the JUnit XML.SMTP_HOST, SMTP_PORT, SMTP_USER, SMTP_PASSWORD, SMTP_FROM, SMTP_TO, SMTP_TLS
Jira / XrayImports the JUnit into an Xray test execution (Cloud REST v2).XRAY_CLIENT_ID, XRAY_CLIENT_SECRET, XRAY_PROJECT_KEY (XRAY_BASE_URL)

Trigger from the Reports page (Email / Xray buttons) or POST /api/reports/{id}/publish. Adding another ALM (Zephyr, TestRail) is a new ResultPublisher subclass — nothing else changes.

Scheduling

APScheduler runs targets on a timer — once, daily or weekly, with a start time and (for endurance) stop conditions. The scheduler starts the run; the engine enforces the end. Manage schedules from the Scheduler page.

Configuration & tools

Targets & the credential vault

Add a Target per device you talk to (SSH host, ADB device, serial port, camera, power/DLT/ETFW script). Passwords and keys are stored in an AES-256-GCM vault and never written in plain text; they're decrypted only at run time and injected into the bound step.

ffmpeg resolution (camera / RDP)

  1. an explicit ffmpeg_path on the step, else
  2. ffmpeg on PATH (winget install Gyan.FFmpeg), else
  3. for SSH camera/RDP, turbossh auto-downloads and caches ffmpeg under ~/.turbossh/ffmpeg/ (and pushes it to the remote box).

adb resolution

turboadb finds a system adb or fetches platform-tools on demand; set adb_path to pin one.

Install & run

Maestro ships as a single PyPI wheel (backend + bundled UI, no binaries — a few MB).

# from PyPI (pulls turboadb + turbossh)
pip install maestro-automation
maestro                         # starts the server and opens the browser

# self-contained folder install (own venv + data + launcher)
maestro-setup

# from source
pip install -r requirements.txt
python app.py                  # builds the UI on first run

SQLite by default; point DATABASE_URL at PostgreSQL for a shared server — same code. Fully offline after install (outbound network only for Jira/Xray, email, and networked SSH/RDP devices).

Security

  • Credential vault — AES-256-GCM; set SECRET_KEY to derive the key from your own secret.
  • Access — role-based (admin / engineer / viewer) with an audit log of who changed and ran what.
  • Local mode — zero-config on a laptop (loopback). On a shared server, set MAESTRO_API_TOKEN so every request must present it.
  • Trusted scripts — run_file / run_command execute with your full environment (they're your bench scripts); only inline run_script is sandboxed.
