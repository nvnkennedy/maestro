Maestro documentation

An open-source, offline-capable, no-code test-automation platform for automotive Hardware-in-the-Loop (HIL) and embedded device testing.

Design tests visually, run them against Linux / QNX / Android targets (local or over RDP), and get detailed step-level reports with attachments. Device control lives in your own tools — Maestro only orchestrates them through thin adapters.

Overview

Maestro lets mostly-manual engineers build test cases on a canvas, reuse a large template gallery, and execute them against automotive benches — without writing code. A small generic runner executes each test; reports are step-level with inline logs, screenshots and DLT traces.

Visual, not code

Drag actions onto a React Flow canvas — branches, parallel lanes and endurance loops. Python is an optional escape hatch.

Your tools

Adapters wrap turboadb, turbossh and your bench scripts. Maestro never reimplements adb/ssh/power.

Real reports

Genuine Allure results (with trends) + JUnit XML, with inline evidence on every step.

Offline

Runs identically on a shared server and a bench laptop, fully offline after install.

Design principles

  1. Manual-engineer first. The primary path is a visual canvas + template gallery; writing Python is never required.
  2. Scenarios are data. A test case is a versioned list of steps a single generic runner executes — not per-test Python files.
  3. One source of truth per capability. Device logic lives in your tools; Maestro only calls them through thin adapters.
  4. Drop-in extensibility. Any script becomes a runnable step with zero framework changes; new adapters auto-discover from a manifest.
  5. Open-source only. No paid licenses or internet at run time (Jira/Xray and email are optional outbound integrations).

Architecture

One React single-page app over a FastAPI backend, with a pluggable adapter layer that drives the bench.

FrontendDesigner, management screens, live run, reports — served by the backend in production.React 18 · Vite · TypeScript · Tailwind · React Flow
Backend / APIPersistence, ownership, scheduling, run orchestration, live WebSocket events, integrations.FastAPI · SQLAlchemy 2 · Pydantic v2 · APScheduler

A test step is just data: { "action": "ssh.execute_command", "parameters": {…}, "timeout_seconds": 30 }. Each adapter exposes named actions returning a uniform AdapterResult.

Visual designer

A real React Flow graph canvas (the default Flow view; a Classic canvas and List view are also available). Pick a Suite → Scenario → Test case, then drag actions from the categorized, searchable palette.

Nodes

  • Each node is one step: action chip, label, timeout chip, and a "no target" warning if a device binding is missing.
  • Add by dragging from the palette or "Add node"; edit by double-clicking; hover for edit / branch / delete.
  • Drag a node up/down to change run order; Tidy auto-arranges; a minimap + zoom keep big flows readable.

Run order & targets

Run order is derived from the canvas layout (top → bottom). A step can be bound to a configured device target (its decrypted credentials are injected at run time) or left local. The Run on selector chooses Local or a saved RDP machine.

Branches, parallel & loops

Branches (Yes/No) — a node's branch button turns it into a decision: it checks the previous step's output for a string and routes accordingly. Parallel — group adjacent steps to run concurrently, then rejoin. Endurance loops — repeat a whole case for N cycles (see Endurance).

KnobEffect
_loopRepeat a single step N times (all iterations must pass).
_if{source_step, contains, skip_to, else_skip_to} — jump based on an earlier step's output.
_parallel_groupConsecutive steps sharing a group name run concurrently.
_always_runCollector steps (logs/screenshots) that run even after an earlier failure.
{{steps.N.output}}Substitute a previous step's output into a string parameter.

Run your own scripts

No special folder, no framework change. Keep scripts where they live and point a step at them. Output streams live to the run console; produced files attach to the report.

StepUse it for
system.run_fileAny script/exe by path, with your own args.
system.run_commandAny full command line (through the shell).
system.run_registeredA script registered once under Templates → Scripts, run by script_id.
system.run_scriptInline Python/PowerShell you paste in (sandboxed).

Interpreter auto-detected by extension

.ps1 → powershell -File   .bat/.cmd → cmd /c   .py → python   .sh → bash   .exe → run directly

Parameters (all optional)

{
  "path": "C:/bench/power_control.py",
  "args": ["normal_power_cycle", "--port", "COM5"],   // list or string
  "cwd": "C:/bench", "env": { "TARGET": "ECU1" }, "timeout": 120,
  "expect_contains": "OK",      // pass/fail on stdout (match_mode: contains|wildcard|regex)
  "attach_output": true,
  "match_file": "C:/bench/out/run.dlt", "match_pattern": "BootComplete"  // pass only if regex found
}

The bench-tool adapters (power.* / dlt.* / etfw.*) are the same idea with fixed verbs — set script_path (+ com_port / channel / extra_args). With no script they simulate, so test design works without a bench.

Custom adapter plugins

Turn a script into a reusable action available everywhere — drop a folder under data/plugins/<name>/ with a manifest and an adapter:

# manifest.json
{ "name": "myscript", "version": "1.0", "entry_point": "adapter:MyScriptAdapter" }

# adapter.py
class MyScriptAdapter(BaseAdapter):
    name = "myscript"
    def _register_actions(self): self.actions = {"do_thing": self._do_thing}
    async def _do_thing(self, params):
        return await self._run_subprocess(["python", r"C:\x.py"], timeout=60)

Restart or use Plugins → Reload. Your actions appear as myscript.do_thing, ready to drop on the canvas.

Power · DLT · ETFW

Thin wrappers around your bench scripts. Set script_path on the device config; with none set they simulate.

DLT

ETFW

bus_sleep_on/off, set_state <state>, get_state — the last state is tracked for assertions.

Serial

Camera (local)

Reports — Allure & JUnit

Every run produces an HTML report with collapsible step cards, parameters, output, retries and inline attachments (screenshots embedded, logs/DLT downloadable). Beyond that, Maestro exports the real interchange artifacts:

  • Real Allure results — GET /reports/{id}/allure downloads an allure-results bundle; run allure serve <dir> for the genuine Allure UI with trends/history across runs.
  • JUnit XML — GET /reports/{id}/junit for CI dashboards and Jira/Xray import.
  • Server-rendered Allure HTML — GET /reports/{id}/allure-html if the Allure CLI is installed.

From the Reports page: open/download a report, download Allure results or JUnit, compare two runs, bulk-delete, re-run.

Endurance / stability

Run a whole test case repeatedly to surface intermittent failures. Set a cycles count in the Run menu (or cycles + stop_conditions on POST /executions).

Stop conditionMeaning
max_durationStop after this many seconds of wall-clock time.
consecutive_failuresStop after N cycles fail in a row.
failure_thresholdStop after N total failed cycles.

Each cycle records a CycleResult; the report shows a roll-up — per-cycle pass/fail chips, totals and the first-failure cycle. GET /reports/{id}/cycles returns the raw data. Pairs naturally with Allure trends.

Scheduling

APScheduler runs targets on a timer — once, daily or weekly, with a start time and (for endurance) stop conditions. The scheduler starts the run; the engine enforces the end. Manage schedules from the Scheduler page.

Configuration & tools

Targets & the credential vault

Add a Target per device (SSH host, ADB device, serial port, camera, power/DLT/ETFW script). Passwords/keys live in an AES-256-GCM vault, decrypted only at run time and injected into the bound step.

ffmpeg (camera / RDP)

  1. an explicit ffmpeg_path on the step, else
  2. ffmpeg on PATH (winget install Gyan.FFmpeg), else
  3. for SSH camera/RDP, turbossh auto-downloads + caches ffmpeg under ~/.turbossh/ffmpeg/ (and pushes it to the remote).

adb

turboadb finds a system adb or fetches platform-tools on demand; set adb_path to pin one.

Install & run

Maestro ships as a single PyPI wheel (backend + bundled UI, no binaries — a few MB).

# from PyPI (pulls turboadb + turbossh)
pip install maestro-automation
maestro                         # starts the server, opens the browser

# self-contained folder install (own venv + data + launcher)
maestro-setup

# from source
pip install -r requirements.txt && python app.py

SQLite by default; point DATABASE_URL at PostgreSQL for a shared server — same code. Fully offline after install (outbound only for Jira/Xray, email, networked SSH/RDP devices).

Security

  • Credential vault — AES-256-GCM; set SECRET_KEY to derive the key from your own secret.
  • Access — role-based (admin / engineer / viewer) with an audit log of who changed and ran what.
  • Local mode — zero-config on a laptop (loopback). On a server, set MAESTRO_API_TOKEN so every request must present it.
  • Trusted scripts — run_file/run_command use your full environment (they're your bench scripts); only inline run_script is sandboxed.
